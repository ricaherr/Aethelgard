# 📜 AI ORCHESTRATION RULES: AETHELGARD SYSTEM

## 🎯 1. FILOSOFÍA Y SSOT
- **Arquitectura:** Hexagonal. El Core debe ser agnóstico.
- **Soberanía de Persistencia (Regla de ORO):** SSOT única: datos residen en BD, no en archivos JSON (excepto seed data idempotente).
- **Separación Global-Tenant (Regla de ORO)**: Arquitectura Híbrida Unificada:
  - `sys_*` tablas: Global, Admin-managed, Trader read-only
  - `usr_*` tablas: Personal per trader, aislado por UUID, soberanía del trader
  - **Violación de convención → RECHAZADO por `validate_all.py`**

## 🗄️ 2. REGLAS DE BASE DE DATOS (PERSISTENCIA)

### 2.1 Estructura de BD Global vs Tenant
- **DB Global** (`data_vault/global/aethelgard.db`): Contiene SOLO `sys_*` tablas (auth, memberships, audit, state, strategies, economic_calendar)
- **DB Tenant** (`data_vault/tenants/{uuid}/aethelgard.db`): Contiene SOLO `usr_*` tablas (assets_cfg, trades, signals, strategy_params, credentials, positions)

### 2.2 Creación Única y Persistencia
- **La base de datos por tenant se crea UNA SOLA VEZ.** El archivo `data_vault/tenants/{tenant_id}/aethelgard.db` solo se crea cuando no existe (`provision_tenant_db` se invoca únicamente si `not os.path.isfile(db_path)`). No se vuelve a "crear" en cada petición ni en cada arranque.

### 2.3 Migraciones Aditivas (Schema Evolution)
- **Las migraciones son solo aditivas.** `run_migrations()` añade columnas o filas faltantes (por ejemplo `ALTER TABLE ... ADD COLUMN`, `INSERT ... WHERE NOT EXISTS`). Nunca se borran datos existentes ni se reemplazan claves de `sys_state` que ya tienen valor. Así se preservan operaciones, señales y configuración del usuario.

### 2.4 No Sobrescribir con Defaults
- **Nunca sobrescribir datos existentes con defaults.** Si una clave ya existe en sys_state (ej. max_daily_risk_pct), no persistir un valor por defecto encima. Solo sembrar cuando la clave falta. Los defaults se usan en memoria para respuestas de API si hay error de lectura, pero no se persisten encima de datos ya guardados.

### 2.5 Prohibición de Redundancia
- **Datos de `sys_*` NUNCA se duplican en `usr_*`.** Trader filtra contra `sys_strategies` en runtime, NO copia a `usr_*`.
- **Razón**: Mantener SSOT único nivel (global). Si estrategia global cambia → todos los traders lo ven automáticamente.

## 🏗️ 3. REGLAS DE ARQUITECTURA
- **Aislamiento:** Conectores solo exportan datos normalizados. Trader B NO accede datos Trader A.
- **Asyncio:** Sistema 100% asíncrono y no bloqueante.
- **Inyección de Dependencias:** Obligatoria inyectada desde el MainOrchestrator. Prohibido `self.storage = StorageManager()`.
- **Comunicación:** Uso de WebSockets (JSON) para la comunicación entre el Core y conectores/UI.
- **Delegación de Responsabilidad**: UniversalEngine → consulta `sys_strategies` (global) + filtra `usr_assets_cfg` (personal) → genera `usr_signals`

## 🛠️ 4. CONTROL DE COMPLEJIDAD
- **Límite de Masa (Regla de ORO):** Ningún archivo >30KB o >500 líneas. Si se excede, el Orquestador debe obligar a una división antes de que el Ejecutor proceda.

## 🛠️ 5. PAUTAS DE CODIFICACIÓN Y TRAZABILIDAD
- **Trazabilidad (Regla de ORO):** Obligatorio el uso de Trace_ID en cada operación.
- **Higiene (Regla de ORO):** El Ejecutor debe borrar archivos temporales y logs antes de la entrega.
- **Check Before Create (Regla de ORO):** ANTES de crear cualquier función, tabla, o código:
  1. BUSCAR si ya existe similares o equivalentes
  2. Si existe, refactorizar la existente (DRY - Don't Repeat Yourself)
  3. Si no existe, entonces crear código NUEVO
  4. Prohibido duplicar código, tablas o lógica existente
  5. Si notas duplicación retroactivamente, eliminar la copia e usar la original
- **Auditoría de Nombres (Regla de ORO)**: Script `audit_table_naming.py` verifica que TODAS las tablas usan `sys_*` o `usr_*`.
- **Calidad:** Type Hints obligatorios al 100% y respeto a Black/Mypy.
- **Idioma:** Chat/Documentación en Español. Código y documentación técnica en Inglés.

## ✅ 6. PROTOCOLO DE VALIDACIÓN (QA)
- **Gestión:** El Ejecutor debe mover tareas en SPRINT.md de [TODO] a [IN_PROGRESS] y luego a [QA].
- **Integridad:** Ejecución obligatoria de validate_all.py antes de cualquier Handshake.
- **Naming Compliance**: `audit_table_naming.py` rechaza tablas sin prefijo (sys/usr)

## 🔄 7. EVOLUCIÓN Y DRIFT
- **Feedback Loop:** Las decisiones del Alpha Engine deben contrastarse contra el resultado real a 5, 10 y 20 velas para calibración automática.
- **Arquitectura Agnóstica + Híbrida**: Global servicios compartidos respetan soberanía personal del trader.
